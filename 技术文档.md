# 智能温室环境监测与控制平台技术文档

## 1. 项目概述

### 1.1 项目简介
智能温室环境监测与控制平台是一个基于物联网技术的现代化农业监控系统，实现了对温室环境的实时监测、数据采集、智能控制和远程管理。

### 1.2 核心功能
- 实时环境监测：温度、湿度、光照、土壤湿度等
- 设备自动控制：风扇、加热器、喷淋、遮阳、补光灯等
- 规则引擎：基于阈值和趋势的智能决策
- 手动控制：支持人工干预
- 实时数据推送：WebSocket 实时通信
- 历史数据查询：传感器数据追溯

### 3.3 核心业务逻辑

#### 3.3.1 数据接入流程 (MqttIngestionService)

**功能说明**：负责接收并处理来自 MQTT Broker 的传感器和执行器消息。

**处理流程**：
1. 订阅 MQTT 主题：
   - 传感器主题：`sensor/{greenhouseId}/{sensorType}/{sensorId}`
   - 执行器主题：`actuator/{greenhouseId}/{actuatorType}/{actuatorId}/command`

2. 传感器数据处理 (`handleSensor` 方法)：
   - 解析 JSON 载荷为 `SensorMqttPayload`
   - 调用 `SensorDataService.ingestFromMqtt()` 进行后续处理

3. 执行器消息处理 (`handleActuator` 方法)：
   - 更新执行器状态
   - 记录命令执行历史（如果包含 commandId）
   - 通过 WebSocket 推送状态更新

**关键代码位置**：`backend/src/main/java/com/example/greenhouse/service/MqttIngestionService.java:41`

#### 3.3.2 传感器数据处理 (SensorDataService)

**功能说明**：处理传感器数据的存储、更新和规则触发。

**处理流程**：
1. 查找或创建大棚实体
2. 查找或创建传感器实体
3. 保存历史数据到 `sensor_data` 表
4. 更新传感器的最新状态
5. 更新 Redis 缓存
6. 调用规则引擎进行评估
7. 通过 WebSocket 推送实时数据

**关键代码位置**：`backend/src/main/java/com/example/greenhouse/service/SensorDataService.java:44`

#### 3.3.3 规则引擎 (RuleEngineService)

**功能说明**：基于阈值规则自动控制执行器。

**核心逻辑**：
1. 检查是否启用手动模式，若是则跳过
2. 查找对应指标的规则配置
3. 判断数值是否超出范围：
   - 低于最小值 → 开启对应设备（如加热器）
   - 高于最大值 → 开启对应设备（如风扇）
   - 正常范围 → 关闭相关设备

**执行器映射规则**：
| 指标 | 超出方向 | 执行器类型 |
|------|---------|-----------|
| temperature | 高于上限 | fan（风扇） |
| temperature | 低于下限 | heater（加热器） |
| humidity | 低于下限 | sprayer（喷淋） |
| light | 高于上限 | shade（遮阳） |
| light | 低于下限 | light（补光灯） |

**关键代码位置**：`backend/src/main/java/com/example/greenhouse/service/RuleEngineService.java:39`

#### 3.3.4 实时缓存服务 (RealtimeCacheService)

**功能说明**：使用 Redis 缓存传感器数据，支持时间窗口数据。

**缓存结构**：
- 最新数据：`env:{greenhouseId}:{sensorId}:latest` (Hash)
- 时间窗口数据：`env:{greenhouseId}:{sensorId}:win` (List，最多保留 100 条)

**关键代码位置**：`backend/src/main/java/com/example/greenhouse/service/RealtimeCacheService.java:23`

### 4.3 commandId 详细说明

#### 4.3.1 核心作用
`commandId` 是指令的唯一标识符，主要用于：
1. **指令追踪**：在整个指令流程中保持唯一标识
2. **去重处理**：防止同一指令被重复执行
3. **状态关联**：执行器通过 `commandId` 关联反馈的状态
4. **日志审计**：便于排查问题和调试

#### 4.3.2 生成规则
```java
private String buildCommandId(String actuatorId) {
    return "cmd_" + DateTimeFormatter.ofPattern("yyyyMMddHHmmss")
            .withZone(ZoneId.systemDefault())
            .format(Instant.now())
            + "_" + actuatorId;
}
```

**格式说明**：
- 前缀：`cmd_`
- 时间戳：`yyyyMMddHHmmss`（年月日时分秒）
- 执行器ID：`actuatorId`

**示例**：`cmd_20260302120000_fan_01`

#### 4.3.3 去重逻辑
```java
if (payload != null && payload.getCommandId() != null) {
    boolean exists = actuatorCommandMapper.findByCommandId(payload.getCommandId()).isPresent();
    if (!exists) {
        // 保存命令记录
        ActuatorCommandEntity command = ActuatorCommandEntity.builder()
                .commandId(payload.getCommandId())
                .actuatorId(actuatorId)
                .greenhouseId(greenhouseId)
                .type(actuatorType)
                .action(payload.getAction() == null ? "unknown" : payload.getAction())
                .reason(payload.getReason())
                .timestamp(timestamp)
                .source(ActuatorCommandEntity.Source.AUTO)
                .build();
        actuatorCommandMapper.save(command);
    }
}
```

### 4.4 执行器状态反馈流程

#### 4.4.1 核心功能
执行器通过 `actuator/{greenhouseId}/{actuatorType}/{actuatorId}/status` 主题向平台反馈当前状态，实现双向通信和状态同步。

#### 4.4.2 处理流程
1. 执行器接收指令并执行
2. 执行器发布状态反馈到对应的状态主题
3. 后端 `MqttIngestionService` 接收并解析
4. 更新执行器状态到数据库
5. 通过 WebSocket 推送状态更新到前端

#### 4.4.3 用途
- **指令确认**：反馈指令是否成功执行
- **状态同步**：确保平台了解执行器实际状态
- **故障上报**：上报设备故障或异常
- **性能监控**：上报运行时间、电量等信息

## 6. 数据流程

```

   ↓

   ↓

   ↓

   ├─ 保存到 MySQL (sensor_data)
   ├─ 更新传感器状态
   ├─ 缓存到 Redis
   └─ 触发 RuleEngineService
       ↓
   5. RuleEngineService 评估规则
       ├─ 检查手动模式
       ├─ 匹配规则
       └─ 决定是否控制执行器
           ↓
       6. MqttCommandPublisher 发送指令
           ↓
       7. MQTT Broker 转发
           ↓
       8. 执行器执行动作
           ↓
       9. 执行器发布状态反馈
           ↓
       10. MqttIngestionService 接收状态
           ↓
       11. 更新执行器状态到数据库
           ↓
       12. WebSocket 推送状态更新
```

   5. MQTT Broker 转发

## 7. 部署要求

### 7.1 环境依赖
- JDK 17+
- MySQL 8.0+
- Redis 6.0+
- MQTT Broker (EMQX 或 Mosquitto)

### 7.2 启动顺序
1. 启动 MySQL
2. 启动 Redis
3. 启动 MQTT Broker
4. 启动后端服务
5. 部署前端

### 7.3 数据库初始化
项目使用 JPA 的 `ddl-auto: update`，首次启动会自动创建表结构。

### 7.4 MQTT Broker 配置

#### 7.4.1 EMQX 配置
1. 下载并安装 EMQX
2. 启动 EMQX：`emqx start`
3. 访问控制台：`http://localhost:18083`
4. 默认用户名：`admin`，密码：`public`
5. 配置允许匿名连接（或创建认证用户）

#### 7.4.2 Mosquitto 配置
1. 下载并安装 Mosquitto
2. 编辑 `mosquitto.conf`：
   ```
   listener 1883
   allow_anonymous true
   ```
3. 启动 Mosquitto：`mosquitto -c mosquitto.conf`

### 7.5 端口说明
| 服务 | 端口 | 说明 |
|------|------|------|
| 后端服务 | 8081 | REST API 和 WebSocket |
| MySQL | 3306 | 数据库 |
| Redis | 6379 | 缓存 |
| MQTT Broker | 1883 | MQTT 协议 |
| EMQX 控制台 | 18083 | MQTT 管理界面 |
