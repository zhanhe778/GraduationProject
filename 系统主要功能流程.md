## 系统主要功能流程概述

本系统是一个“温室环境监测与自动控制平台”，整体可以拆分为三条主干流程：

- 环境数据采集与展示流程
- 自动控制（规则引擎驱动执行器）流程
- 人工干预（手动模式与手动控制）流程

下面分别按“从前端到后端”的视角进行梳理。

---

## 一、环境数据采集与展示流程

### 1.1 采集与上传

1. 物理/模拟传感器周期性采集环境数据（温度、湿度、光照等）。
2. 传感器通过 MQTT 客户端（如 MQTTX 或自研模拟程序）向 EMQX 发送消息：
   - Topic 格式：`raw/sensor/{greenhouseId}/{sensorType}/{sensorId}`
   - Payload：JSON，字段包括 `greenhouseId`、`sensorId`、`type`、`value`、`unit`、`timestamp`。

### 1.2 边缘网关过滤

1. 边缘网关进程 `EdgeGatewaySimulator` 连接同一个 EMQX Broker，订阅 `raw/sensor/#`。
2. 对每条原始传感器数据进行本地计算和过滤：
   - 对每个传感器维护最近一次上报值 `lastValue` 和上报时间 `lastReportTime`。
   - 三类规则：
     - 变化阈值：当前值相对上次值变化比例超过 5% 才上报；
     - 限流：两次上报间隔至少 1 秒，避免抖动；
     - 心跳：即使数值变化不大，只要超过 60 秒未上报也强制上报一次。
3. 只有满足“需要上报”的数据才会被重新发布到新的 Topic：
   - Topic：`sensor/{greenhouseId}/{sensorType}/{sensorId}`
   - Payload：仍然是原始 JSON 内容。

### 1.3 后端接入与存储

1. Spring Boot 后端通过 Spring Integration MQTT 订阅 `sensor/+/+/+`。
2. `MqttIngestionService.handleMessage(...)` 根据 Topic 前缀识别为传感器数据，调用 `handleSensor(...)`：
   - 解析 Topic 中的 `greenhouseId`、`sensorType`、`sensorId`；
   - 反序列化 Payload 为 `SensorMqttPayload`。
3. 调用 `SensorDataService.ingestFromMqtt(...)` 完成以下逻辑：
   - 确保 `greenhouses` 表中存在该大棚，没有则创建；
   - 确保 `sensors` 表中存在该传感器，没有则创建；
   - 将该条读数作为历史数据插入 `sensor_data` 表；
   - 更新 `sensors` 表中对应传感器的 `lastValue` 与 `lastTimestamp`；
   - 将数据交给规则引擎进行后续自动控制（见下一节）。

### 1.4 前端实时展示

1. 后端在关键数据变更时通过 WebSocket 推送 `RealtimeEvent` 到前端：
   - Topic：`/topic/greenhouse/{greenhouseId}`
   - 内容包括最新传感器状态、执行器状态等。
2. 前端单页应用（SPA）订阅 WebSocket Topic，并根据事件类型更新界面：
   - 仪表盘展示各传感器当前值（温度、湿度、光照等）；
   - 大棚详情界面展示每个传感器、执行器的实时状态。
3. 用户也可以通过前端页面分页查询历史数据：
   - 前端调用后端 REST 接口；
   - 后端通过 `sensor_data` 表分页查询指定大棚/传感器的历史记录并返回。

## 二、自动控制（规则引擎驱动执行器）流程

### 2.1 规则配置

1. 用户在前端的“规则配置”界面，为每个大棚配置各个指标的阈值范围：
   - 指标名称：如 `temperature` / `humidity` / `light`；
   - 最小值 `minValue`、最大值 `maxValue`；
   - 是否启用 `enabled`。
2. 前端通过 REST 接口提交配置；
3. 后端将配置持久化到 `rule_ranges` 表中：
   - 一条记录对应一个大棚 + 一个指标。

### 2.2 规则评估入口

1. 每当 `SensorDataService.ingestFromMqtt(...)` 写入一条新数据时，会调用 `RuleEngineService.evaluate(...)`。
2. `evaluate(...)` 先检查：
   - 若该大棚处于手动模式（`manual_modes.enabled = true`）则直接返回，不进行自动控制；
   - 否则继续读取该大棚 + 指标在 `rule_ranges` 中的配置。

### 2.3 超限判断与执行器选择

1. 对当前传感器值 `value` 与规则的 `minValue` / `maxValue` 比较：
   - 若 `value < minValue`：认为“低于下限”，触发低限逻辑；
   - 若 `value > maxValue`：认为“高于上限”，触发高限逻辑；
   - 否则：认为“恢复正常”，执行回正常逻辑。
2. 根据指标类型和超限方向，通过 `resolveActuatorType(metric, aboveMax)` 映射到具体执行器类型：
   - 温度 `temperature`：
     - 高温（超过上限）：开启风扇 `fan`；
     - 低温（低于下限）：开启加热器 `heater`；
   - 湿度 `humidity`：
     - 低湿：开启喷淋 `sprayer`；
   - 光照 `light`：
     - 光照过强：开启遮阳 `shade`；
     - 光照不足：开启补光灯 `light`。

### 2.4 互斥控制与命令下发

1. 当需要“开启某类设备”时，`handleOutOfRange(...)` 会：
   - 对温度类指标，先关闭另一方向的设备（例如高温时关闭 heater，低温时关闭 fan），保证互斥；
   - 查询 `actuators` 表中对应类型设备的列表，选择一个进行控制；
   - 若该设备当前状态已为 `on`，则不重复发命令。
2. 构建并保存命令记录：
   - 在 `actuator_commands` 表中插入一条记录，包含 `commandId`、`actuatorId`、`action`、`reason`、`source=AUTO` 等；
   - 更新 `actuators` 表中对应设备的 `status` 和 `lastCommandAt`。
3. 下发命令：
   - `MqttCommandPublisher` 根据设备类型构造 Topic，如 `actuator/{greenhouseId}/{type}/{actuatorId}/command`；
   - 将命令内容以 JSON 形式通过 MQTT 发送到执行器侧。
4. 通过 WebSocket 将执行器最新状态推送给前端，前端界面实时体现风扇/加热器等的开关变化。

### 2.5 数值恢复正常时的处理

1. 当传感器值回到阈值范围内时，`evaluate(...)` 会调用 `handleBackToNormal(...)`：
   - 对应指标下，关闭因高限开启的设备（例如风扇）；
   - 同时关闭因低限开启的设备（例如加热器）；
2. 同样会：
   - 在 `actuator_commands` 中记录关闭命令；
   - 更新 `actuators` 状态为 `off`；
   - 通过 MQTT 下发指令，并通过 WebSocket 通知前端。

## 三、人工干预功能流程

### 3.1 手动模式开关流程

1. 前端在大棚详情页提供“手动模式”开关。
2. 用户切换开关时，发送 REST 请求到后端 `ManualModeService`：
   - 若开启手动模式：在 `manual_modes` 表中将对应大棚的 `enabled` 设为 `true`，并更新 `updatedAt`；
   - 若关闭手动模式：设置 `enabled = false`。
3. 规则引擎在 `evaluate(...)` 中会优先检查：
   - 若某大棚 `manual_modes.enabled = true`，则**跳过所有自动控制逻辑**，只记录数据不发控制命令。
4. 前端依据该状态在界面上标记当前大棚处于“手动”还是“自动”模式。

### 3.2 手动控制执行器流程

1. 在前端“手动控制”界面，用户可以针对某个执行器（如风扇、加热器）点击“开启”或“关闭”。
2. 前端发送手动命令请求到后端 `ManualControlService`，携带：
   - greenhouseId、actuatorId、目标 action（on/off）、原因（可选备注）。
3. 后端逻辑：
   - 更新 `actuators` 表中对应执行器的 `status` 和 `lastCommandAt`；
   - 在 `actuator_commands` 表中插入一条 `source=MANUAL` 的命令记录；
   - 通过 `MqttCommandPublisher` 下发 MQTT 指令；
   - 通过 WebSocket 推送最新执行器状态给前端。
4. 前端界面实时显示手动操作结果，也可以在“命令历史”里看到对应的手动命令记录。

## 四、查询与展示辅助流程

### 4.1 大棚列表与详情

1. 前端在“首页/大棚列表”界面调用后端 `GreenhouseQueryService.listGreenhouses()`：
   - 查询 `greenhouses`、`sensors`、`actuators` 等表，汇总出每个大棚的概览信息；
   - 返回每个大棚的基础信息与当前状态摘要。
2. 点击某个大棚进入详情页时，调用 `getDetail(greenhouseId)`：
   - 查询指定大棚的：
     - 大棚基础信息；
     - 当前所有传感器状态；
     - 当前所有执行器状态；
     - 已配置的阈值规则；
     - 当前手动模式状态；
   - 以一个聚合的 DTO 返回给前端，作为详情页初始数据。

### 4.2 历史数据查询

1. 前端在“历史曲线/历史记录”界面，根据用户选择的大棚、传感器以及时间范围，调用后端历史查询接口。
2. 后端在 `sensor_data` 表中按：
   - `greenhouseId` + `sensorId`
   - `reading_time` 范围
   - 分页参数
   进行查询，并转换为前端需要的 DTO。
3. 前端将返回的数据绘制为折线图或列表，用于分析某一天或某一时间段的环境变化趋势。

## 五、小结：主干功能一览

综合上述流程，可以将系统的主要功能主线概括为：

1. **数据采集链路**：传感器 → EMQX → 边缘网关过滤 → EMQX → 后端 MQTT 接入 → 数据入库 + 实时状态更新。
2. **自动控制链路**：新数据入库 → 规则引擎判断超限 → 选择执行器 → 互斥控制 → 生成命令记录 → MQTT 下发指令 → WebSocket 同步前端。
3. **人工干预链路**：用户切换手动模式或直接对执行器下发命令 → 更新数据库状态与命令历史 → MQTT 下发指令 → WebSocket 更新界面。
4. **查询展示链路**：前端通过 REST 获取大棚列表、详情、历史数据，结合 WebSocket 实时事件，完成对整个温室系统的可视化展示与控制。
